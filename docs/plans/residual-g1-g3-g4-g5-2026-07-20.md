## 1. 问题陈述与证据

### 1.1 共同背景

§12 hygiene（文档漂移 / package.json / 主题测试 / zip）已收口。下列四项**不是漏做**，而是：

- 平台硬限（G3/G4）  
- 云端不可复现的本机依赖（G1 全量）  
- 已知韧性缺口但有绕过（G5）

### 1.2 分项证据

| ID | 证据 | 用户/维护者痛感 |
|----|------|----------------|
| G1 | 仓内无 `.github/`；门禁依赖本机 `doctor` / smoke / `test:themes` | 外部贡献者/未来自己改 PR 时无强制绿勾 |

### 1.3 非目标（本规划全局）

- 不把 Quiet 回归改成交互 MessageBox（破坏自动化）

## 2. 决策框架（怎么比）

每项方案用同一套维度打分（1–5，越高越好）：

| 维度 | 含义 |
|------|------|
| **用户价值** | 终端用户是否更少踩坑 |
| **维护者价值** | 发版/PR/排障是否更省心 |
| **可行性** | 技术/OS/权限是否允许 |
| **成本** | 实现 + 长期维护（5=低成本） |
| **风险** | 引入回归/误报/范围蔓延（5=低风险） |
| **可逆性** | 做错了能否轻易撤回（5=易逆） |

**总分** = 六维之和（满分 30）。同分看「可行性 × 风险」否决项。

## 3. G1 — 全量 CI / 自动化门禁

### 3.1 问题

本机有 `doctor` + smoke + `npm run test:themes`，但 **PR / push 无强制流水线**。全量 doctor 依赖：Windows + 本机 Store Codex + 可选 live CDP + 安装态路径——云 runner 默认不具备。

### 3.2 方案对比

| 方案 | 描述 | 用户 | 维护 | 可行 | 成本 | 风险 | 可逆 | **Σ** |
|------|------|:----:|:----:|:----:|:----:|:----:|:----:|:----:|
| **G1-A 维持现状** | 只文档写「本机门禁」 | 1 | 2 | 5 | 5 | 5 | 5 | **23** |
| **G1-B 轻量 CI** | GitHub Actions：Node 20 · `npm run test:themes` · 可选 `node --check` 关键 mjs | 1 | 5 | 5 | 4 | 5 | 5 | **25** |
| **G1-C 中量 CI** | B + 主题 import dry-run（无写用户目录）+ schema 扫 11 套 `loadTheme` | 1 | 5 | 4 | 3 | 4 | 4 | **21** |
| **G1-D 全量 Windows CI** | windows-latest 装依赖并尝试 doctor/smoke | 2 | 3 | **1** | 1 | 2 | 2 | **11** |
| **G1-E 仅 pre-commit 本地钩子** | husky/lefthook 跑 test:themes | 1 | 3 | 4 | 3 | 4 | 4 | **19** |

#### 方案细节

**G1-A**  
- 优点：零工作。  
- 缺点：PR 可合并损坏的 schema；与「有 test:themes 却不用」矛盾。

**G1-B（推荐）**  
```yaml
# 示意 .github/workflows/themes-gate.yml
on: [push, pull_request]
jobs:
  themes:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20" }
      - run: npm run test:themes
```
- 优点：与现有脚本零分叉；Linux 即可；失败即红。  
- 缺点：不覆盖 CDP/injector/PS。  
- **不假装** doctor 已在 CI。

**G1-C**  
- 在 B 上对 `themes/*` 批量 `loadTheme`（测试脚本扩展）。  
- 优点：抓坏图路径/坏 json。  
- 缺点：需保证 CI 有 hero 二进制（已在仓内）。

**G1-D**  
- 否决理由：无 Codex 安装 → doctor 噪声；密钥/Store 许可不可用；维护成本高、信号差。

**G1-E**  
- 优点：本地即时。  
- 缺点：可绕过；协作者未装钩子则无效；与 CI 互补而非替代。

### 3.3 G1 推荐

| 项 | 值 |
|----|-----|
| **选定** | **G1-B**，可选随后 **G1-C** 扩 `test:themes` 扫全仓 11 套 |
| **不做** | G1-D |
| **验收** | PR 改坏 `theme-schema.mjs` 或测试用例 → Actions 红；绿时 `test:themes` 本地一致 |
| **回滚** | 删 workflow 文件即可 |

### 3.4 G1 任务拆解（若执行）

1. 新增 `.github/workflows/themes-gate.yml`  
2. README / PROJECT §9 门禁表加「CI：test:themes」  
3. （可选）扩展 test 扫 `themes/*/theme.json` 全部 `loadTheme`  
4. 故意 break 一次验证红灯，再 revert

## 5. G4 — 商店磁贴 / AUMID 裸启（#21）

### 5.1 问题

用户点击**微软商店 Codex 磁贴**时走 package activation（`OpenAI.Codex_…!App`），**无法**被第三方改写为带 `--remote-debugging-port=9335` 的命令行。结果：裸启、无皮肤 → 误判产品故障。

已有缓解：

- 日常入口：`CodexFastLaunch.exe`（AUMID `CodexDreamSkin.FastLaunch`）  
- open 路径发现裸 Codex 时安静 reattach / 必要时重启带 CDP  
- usage / dual-open 文档禁止商店磁贴

### 5.2 方案对比

| 方案 | 描述 | 用户 | 维护 | 可行 | 成本 | 风险 | 可逆 | **Σ** |
|------|------|:----:|:----:|:----:|:----:|:----:|:----:|:----:|
| **G4-A 文档 only** | 维持 usage + dual-open + PAIN | 3 | 5 | 5 | 5 | 5 | 5 | **28** |
| **G4-A+ 文档 + 既有 reattach 可观测** | A + doctor/`dailyEntry` 文案强调；可选首次裸启气球 | 4 | 4 | 5 | 4 | 4 | 5 | **26** |
| **G4-B 开机/定时扫描裸启并 reattach** | 计划任务轮询 | 3 | 2 | 4 | 2 | 2 | 3 | **16** |
| **G4-C 尝试改写/伪装商店 AUMID** | 劫持包激活 | 2 | 1 | **0** | 1 | 1 | 1 | **6** |
| **G4-D 卸载引导去钉商店** | 安装时弹「请取消钉选商店」 | 2 | 3 | 3 | 3 | 2 | 4 | **17** |

### 5.3 关键级说明

**G4-C 不可行（架构否决）**

- Store 包激活由 OS 与包身份绑定。  
- 伪造 AUMID / 注入包清单 = 破坏签名与商店政策，且通常无效。  
- 与产品「不改 asar / 不碰签名包」硬约束冲突。

**G4-B 成本高**

- 常驻扫描 ≈ 第二守护心智负担；与「单 injector」叙事冲突（即使用户无感）。  
- 误杀/误重启 Codex 风险。

**G4-A+（推荐增强）**

- 不新增常驻。  
- 确保 open / check 路径在「裸 Codex」时：  
  1. 已有 reattach 逻辑继续  
  2. 文档 + 可选一次性质气球：「请用任务栏 Codex，不要用商店磁贴」  
- doctor `dailyEntry` 已指向 CodexDreamSkin，保持即可。

### 5.4 G4 推荐

| 项 | 值 |
|----|-----|
| **选定** | **G4-A** 为底线；体验向可选 **G4-A+**（仅文案/一次性提示，无新守护） |
| **永久否决** | G4-C |
| **验收** | 新用户读 usage 30 秒懂「钉任务栏」；点商店磁贴仍可能裸启（诚实预期） |
| **成功指标** | 相关客服/自用误报下降；**不是**「商店磁贴也有皮肤」（不可达） |

## 6. G5 — post-update Quiet 长挂

### 6.1 问题（根因级）

`scripts/windows/publish-runtime.ps1` 尾部：

```powershell
$pPost = Start-Process -FilePath 'powershell.exe' -ArgumentList @(
  '-NoProfile','-ExecutionPolicy','Bypass','-File',$post,'-Quiet','-Repair'
) -Wait -PassThru -WindowStyle Hidden
```

- **`-Wait` 无超时** → 子进程卡在 `check-and-fix -Quiet` 链路时，publish 整体挂死。  
- `post-update-regression.ps1 -Repair` 在 drift 时再 `Start-Process` 调 `check-and-fix -Quiet`（同样可慢）。  
- **Install-Product** 已改为 **soft reattach**（杀旧 → 启新 injector → 补 state），刻意避开 hang-prone 路径。  
- 不对称：产品安装路径已加固，**开发 publish 路径仍暴露**。

### 6.2 方案对比

| 方案 | 描述 | 用户 | 维护 | 可行 | 成本 | 风险 | 可逆 | **Σ** |
|------|------|:----:|:----:|:----:|:----:|:----:|:----:|:----:|
| **G5-A 继续观察** | 文档写「可能慢」；不改代码 | 2 | 2 | 5 | 5 | 4 | 5 | **23** |
| **G5-B 去掉 publish 的 -Repair** | 只刷报告，不 reattach | 2 | 3 | 5 | 5 | 3 | 5 | **23** |
| **G5-C 硬超时 + 超时 soft reattach** | Start-Process 限时（如 45–60s）；超时 Kill + 走 Install 同款 soft reattach | 4 | 5 | 5 | 4 | 4 | 4 | **26** |
| **G5-D publish 全面改 soft reattach** | 与 Install 共用函数，不再调 post-update -Repair | 4 | 5 | 4 | 3 | 4 | 3 | **23** |
| **G5-E check-and-fix 内部硬超时** | 所有调用方受益 | 4 | 4 | 4 | 2 | 3 | 3 | **20** |

#### 方案细节

**G5-A**  
- 适合「极少 publish」；与「发版检查清单」体验冲突。

**G5-B**  
- 优点：立刻消挂。  
- 缺点：publish 后 injector 可能仍跑旧 runtime → smoke/fresh 假绿或假红；曾为此加 `-Repair`。

**G5-C（推荐）**  
1. `publish-runtime.ps1` 对 post-update 使用带超时等待（例：60s）。  
2. 超时：`Stop-Process` 子树 + Warning。  
3. Fallback：调用与 Install 相同的 soft reattach（可抽 `Invoke-CodexSkinSoftReattach` 到 core-win 或 publish 内联复制最小集）。  
4. 仍写 `post-update-report.json` stamp（现有逻辑保留）。  
- 优点：保留 Repair 意图 + 有上界 + 与 Install 收敛。  
- 缺点：需测「正常 <60s」与「超时路径仍 fresh」。

**G5-D**  
- 更干净的长期态；改动面大于 C。可作为 C 的第二步重构。

**G5-E**  
- 正确但触及 check-and-fix 多条路径；回归面大；适合单独 PR。

### 6.3 G5 推荐

| 项 | 值 |
|----|-----|
| **选定** | **G5-C**（publish 超时 + soft reattach fallback） |
| **后续** | 稳定后考虑 G5-D 抽公共函数；G5-E 另议 |
| **验收** | ① 正常 publish 尾部 ≤60s 完成且 `injectorPathFreshness.fresh=true` ② 人为卡住子进程时 publish 不无限等待并 Warning ③ Install 路径行为不变 |
| **SLO（维护者）** | publish 端到端 P95 < 120s（含拷贝）；post 阶段 P95 < 60s |

### 6.4 G5 任务拆解（若执行）

1. 在 `publish-runtime.ps1` 实现 `Wait-ProcessWithTimeout`（或等价）  
2. 超时分支 soft reattach（对齐 Install-Product 语义）  
3. 日志：`post-update exit=` / `post-update timeout → soft reattach`  
4. 手工：publish 两次（热路径 + 可选模拟超时）  
5. CHANGELOG / PAIN 记「G5 加固」一句
